The **WebSocketSender** broadcasts decimated, filtered seismic data in real-time. It transforms the high-frequency 100 Hz raw stream into a lightweight 25 Hz stream optimized for web-based visualization without overwhelming browser rendering engines.

## Architecture

This thread acts as an asynchronous broadcast server, handling multiple concurrent client connections.



- **Input:** `websocket_queue` (100 Hz Raw ADC counts).
- **Processing:** Sliding window filtering and decimation.
- **Output:** JSON payloads via `ws://0.0.0.0:8765`.

---

## Signal Processing Pipeline

To provide a clean "seismic trace" on the dashboard, the data undergoes four stages of processing before broadcast:

<Steps>
1. **Buffering:** Samples are collected into a **5-second sliding window** (500 samples).
2. **Filtering:** A 2nd-order Butterworth bandpass filter (**0.2–10 Hz**) is applied to remove DC offset and high-frequency electronic noise.
3. **Decimation:** The signal is downsampled by a factor of 4 using an anti-aliasing filter, resulting in a **25 Hz** stream.
4. **Packetization:** The most recent 1 second of processed data (25 samples) is wrapped in a JSON envelope and broadcast to all connected clients.
</Steps>



<Aside type="note" title="Why a 5-second window?">
  Digital filters (specifically IIR filters like Butterworth) require a "settling time" to reach stability. By using a 5-second window, we ensure the center 1-second segment being broadcast is free from filter artifacts and "ringing" at the edges.
</Aside>

## Message Specification

The server broadcasts a standardized JSON payload. Clients use the `type` field to distinguish between waveform data and State-of-Health (SOH) updates.

### Sample Payload Example


| Field | Type | Description |
| :--- | :--- | :--- |
| `type` | `int` | `0` = Waveform Data, `1` = SOH/Status. |
| `payload.channel` | `string` | The SEED channel code (e.g., EHZ, EHN). |
| `payload.fs` | `float` | The actual sampling rate after decimation (25.0). |
| `payload.data` | `float[]` | Array of 25 filtered samples (1 second of data). |

## Performance & Scalability

The WebSocket thread is built on `asyncio` to handle high concurrency with minimal CPU overhead.

### Resource Impact
- **Latency:** Total processing time from Queue to Network is **~5ms**.
- **Bandwidth:** Each client consumes only **~2.4 kbps**. 
- **Capacity:** A Raspberry Pi 4 can easily sustain **100+ concurrent dashboard clients** without impacting data acquisition.

## Troubleshooting

### No data on the Dashboard
1. **Service Status:** Verify the WebSocket thread is running.
2. **Port Check:** Ensure port `8765` is open: `sudo ufw allow 8765`.
3. **Browser Console:** Open Chrome DevTools (F12) and check the "Network" tab under "WS" to see if the connection is `Switching Protocols` or returning `404`.

### Dashboard "Jumpy" or Lagging
- **System Load:** High CPU usage (e.g., during an MSeed merge) can cause minor jitter. Check load with `top`.
- **Decimation:** If the client network is extremely slow, increase the `decimation_factor` to `8` (12.5 Hz) in `config.yml` to reduce bandwidth by half.
